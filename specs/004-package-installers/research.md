# Research: Package Installer Support

**Feature**: 004-package-installers  
**Date**: 2026-03-07

## R-001: GoReleaser Homebrew Tap Publishing

**Decision**: Use GoReleaser `brews` section to auto-publish to `infobloxopen/homebrew-tap`

**Rationale**: 
- GoReleaser natively supports `brews` — on tagged release, it clones the tap repo, creates/overwrites `Formula/apx.rb`, commits, and pushes
- The tap repo can start **completely empty** — GoReleaser creates the `Formula/` directory and `.rb` file on first release
- No registration with Homebrew core needed — a tap is just a git repo
- Users install via `brew install infobloxopen/tap/apx` (Homebrew auto-expands `org/tap` to `org/homebrew-tap`)
- Requires a PAT (`HOMEBREW_TAP_TOKEN`) with `contents: write` on the tap repo since it's a cross-repo push
- The existing `.goreleaser.yml` already has a correct `brews` section — it just needs the `token` field and the target repo to exist

**Alternatives considered**:
- Homebrew Core submission: Requires review process, minimum star/popularity threshold, and registration with a third party. Rejected per user constraint.
- `homebrew_casks` (GoReleaser v2.10+): Casks are for GUI apps with `.dmg`/`.pkg` installers, not CLI binaries. `brews` (formulas) is correct for CLI tools.

## R-002: GoReleaser Scoop Bucket Publishing

**Decision**: Use GoReleaser `scoops` section to auto-publish to `infobloxopen/scoop-bucket`

**Rationale**:
- GoReleaser v2 has a `scoops` (plural) section that generates JSON manifests and pushes to a bucket repo
- A Scoop bucket is just a GitHub repo with JSON files — **no central registry, no registration**
- Manifests must be in the **root directory** of the repo (not a subdirectory) for `scoop bucket list` to work
- The repo can start empty — GoReleaser creates the JSON manifest on first release
- Users install via: `scoop bucket add infobloxopen https://github.com/infobloxopen/scoop-bucket && scoop install infobloxopen/apx`
- Requires a PAT (`SCOOP_BUCKET_TOKEN`) or the same token if permissions allow

**Alternatives considered**:
- Scoop Main bucket: Central registry, requires submission. Rejected per user constraint.
- Chocolatey: Requires NuGet packaging and submitting to chocolatey.org. Rejected per user constraint.

## R-003: winget Support

**Decision**: **Do NOT support winget** in this feature

**Rationale**:
- winget requires submitting manifests to `microsoft/winget-pkgs` — a third-party central repository with a review process
- While GoReleaser has a `winget` section, it works by opening PRs to `microsoft/winget-pkgs` (or a fork)
- Self-hosted winget sources exist (`winget source add`) but are impractical — requires running a REST API server, not just a GitHub repo
- This violates the user's constraint: "if anything requires registering with a third party like a central package repository, don't do it"
- Windows users are served by Scoop (P2) and direct GitHub Releases download

**Alternatives considered**:
- GoReleaser winget cross-repo PR: Technically works but requires third-party registration. Rejected.
- Self-hosted winget source: Requires hosting a REST API. Overkill and outside scope.

## R-004: Shell Installer Script

**Decision**: Create `install.sh` at the repo root as a standalone curl-pipe-bash installer

**Rationale**: 
- Universal fallback — works anywhere with curl and bash (CI, Docker, fresh machines)
- Downloads pre-built binaries from GitHub Releases (already generated by GoReleaser)
- Standard pattern: `curl -fsSL https://raw.githubusercontent.com/infobloxopen/apx/main/install.sh | bash`
- Key design decisions:
  - OS/Arch detection via `uname -s`/`uname -m` mapped to Go targets (linux/darwin × amd64/arm64)
  - Version pinning via `VERSION` env var, defaults to latest release
  - Latest version detection via GitHub API redirect (`curl -I .../releases/latest` → parse `Location` header, no rate limiting)
  - SHA256 checksum verification using `checksums.txt` from the release
  - Default install to `~/.local/bin` (user-writable, no sudo), overridable via `INSTALL_DIR`
  - Wrap everything in `main()` for safe `curl | bash` execution
  - Fully non-interactive for CI; respects `NO_COLOR`, `CI` env vars

**Alternatives considered**:
- Godownloader (deprecated, no longer maintained)
- Just linking to GitHub Releases page: Less user-friendly, no checksum verification, more steps

## R-005: Release Workflow

**Decision**: Create `.github/workflows/release.yml` triggered on version tags

**Rationale**:
- Uses `goreleaser/goreleaser-action@v6` (official Action)
- Triggers on `push.tags: ["v*"]` pattern for semantic version tags
- Requires `fetch-depth: 0` for full git history (changelog generation)
- Needs `contents: write` permission for GitHub Releases
- Cross-repo tokens minted at runtime via `actions/create-github-app-token@v1` using the `infobloxopen-release-bot` GitHub App
- GoReleaser handles everything: build → archive → checksum → GitHub Release → Homebrew tap → Scoop bucket
- Single workflow, single GoReleaser invocation — all package manager updates are atomic per release

**Alternatives considered**:
- Separate workflows per package manager: More complex, harder to keep in sync. Rejected.
- Manual release process: Error-prone, doesn't scale. Rejected.

## R-006: Token Strategy

**Decision**: Use the `infobloxopen-release-bot` GitHub App (App ID: 3033530) to mint short-lived tokens at runtime

**Rationale**:
- GitHub Apps provide org-owned, non-personal authentication — no dependency on individual user accounts
- `actions/create-github-app-token@v1` mints a short-lived token scoped to specific repos (`homebrew-tap`, `scoop-bucket`)
- Org secrets `RELEASE_APP_ID` and `RELEASE_APP_PRIVATE_KEY` store the App credentials
- The minted token is passed to GoReleaser as `HOMEBREW_TAP_TOKEN` and `SCOOP_BUCKET_TOKEN` env vars
- The built-in `GITHUB_TOKEN` is scoped to the workflow's repo only — cannot push to other repos
- No token expiry concerns — tokens are freshly minted per workflow run

**Alternatives considered**:
- Fine-grained PATs: Tied to a personal account, expire, and create a bus-factor risk. Rejected.
- Deploy keys: Per-repo SSH keys for push. More secure than PATs but more complex to manage. Rejected in favor of GitHub App.

## R-007: GoReleaser Version

**Decision**: Use GoReleaser v2 (already configured with `version: 2` in `.goreleaser.yml`)

**Rationale**:
- The existing `.goreleaser.yml` already has `version: 2` at the top
- GoReleaser v2 supports `brews` and `scoops` sections
- The `goreleaser/goreleaser-action@v7` action supports v2 via `version: "~> v2"` or `version: latest`
- The Makefile already references `GORELEASER_VERSION := v1.21.2` — this should be updated to v2.x for consistency, but the Action can use its own version
