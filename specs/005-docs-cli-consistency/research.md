### R2: Config Schema Analysis

**Task**: Determine the authoritative config schema and identify mismatches.

**Findings**:

The `Config` struct in `internal/config/config.go`:

```go
type Config struct {
    Version         int                        `yaml:"version"`
    Org             string                     `yaml:"org"`
    Repo            string                     `yaml:"repo"`
    ModuleRoots     []string                   `yaml:"module_roots"`
    LanguageTargets map[string]LanguageTarget   `yaml:"language_targets"`
    Policy          Policy                     `yaml:"policy"`
    Publishing      PublishingConfig           `yaml:"publishing"`
    Tools           ToolsConfig                `yaml:"tools"`
    Execution       ExecutionConfig            `yaml:"execution"`
}
```

Validation rules (`validateConfig`): `version != 0`, `org != ""`, `repo != ""`.

**BUG FOUND**: `internal/schema/app.go` line 98 generates:
```yaml
kind: proto
module: payments.ledger.v1
org: testorg
version: v1
```
- `version: v1` is a string — `Config.Version` expects `int`
- Missing `repo` field — `validateConfig` requires `repo != ""`
- Has `kind` and `module` fields — not in `Config` struct (silently ignored by `yaml.v3` but misleading)

`apx.example.yaml` is correct (has `version: 1`, `org`, `repo`, etc.)

### R3: Lockfile Schema Analysis

**Task**: Determine the lockfile schema and compare to docs.

**Findings**:

```go
type LockFile struct {
    Version      int                          `yaml:"version"`
    Toolchains   map[string]ToolchainLock     `yaml:"toolchains"`
    Dependencies map[string]DependencyLock    `yaml:"dependencies"`
}
type DependencyLock struct {
    Repo    string   `yaml:"repo"`
    Ref     string   `yaml:"ref"`
    Modules []string `yaml:"modules"`
}
```

Current docs show a flat-list format instead of a map-keyed format. Need to update docs.

### R4: Existing Test Infrastructure

**Task**: Assess existing doc-parity tests and determine extension strategy.

**Findings**:

`cmd/apx/commands/doc_parity_test.go` (273 lines) already exists with:
- `TestDocParity_InitCanonical` — validates output patterns + file/dir creation
- `TestDocParity_InitApp` — validates output patterns + file/dir creation
- `TestDocParity_LintCommand` — validates command exists, `Use` string
- `TestDocParity_BreakingCommand` — validates command exists
- `TestDocParity_PublishCommand` — validates command + flags exist
- `TestDocParity_ConsumerCommands` — validates search/add/gen/sync/unlink exist
- `TestDocParity_CompletionCommand` — validates completion works
- `TestDocParity_OverlayPaths` — validates path format convention
- `TestDocParity_GitIgnorePattern` — validates gitignore pattern
- `TestDocParity_CommandExamples` — **HAS BUG**: search example uses 2 args (`payments`, `ledger`) but `MaximumNArgs(1)`

**Test framework**: Standard `go test` (not testscript). Tests instantiate `NewRootCmd("test")`, call `cmd.Find()` or `cmd.Execute()`.

**Extension strategy**: Add tests for:
1. All flag existence on every command (loop over map)
2. Required-flag validation (test that missing `--against` on breaking fails)
3. Config roundtrip test (init → validate)
4. Fix search example to 1 arg

### R5: Documentation Infrastructure

**Task**: Understand the doc build system for broken-link detection.

**Findings**:

- Build: Sphinx + myst_parser + sphinx_design + sphinx_copybutton + sphinx_togglebutton
- Config: `docs/conf.py` (99 lines)
- Format: MyST Markdown (`.md`)
- Admonitions: Use `{admonition}` directive from sphinx_design for callout blocks
- Link checking: `sphinx-build -b linkcheck` detects broken links

For "Planned" markers, can use:
```markdown
```{admonition} Planned — not yet available
:class: warning
This command is on the roadmap but not implemented in the current release.
```

### R8: Broken Toctree References

**Task**: Identify all broken toctree links.

**Decision**: Run `sphinx-build` at implementation time to get the exact list. From the spec audit, approximately 27 broken references exist spanning `docs/index.md` and sub-pages. Most reference sub-pages that were never created. Resolution: create minimal stub pages or remove the references.

### R9: Exit Code Strategy

**Task**: Determine actual exit codes and establish canonical table.

**Findings**:

From `main.go` and cobra behavior:
- **0**: Success
- **1**: General error (command failure, unknown error)
- **6**: Validation failure (lint errors, breaking changes detected)

Most commands return error → cobra exits 1. Some commands use `os.Exit(6)` for validation.

**Decision**: Define a single table in `cli-reference/index.md`:

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | General error |
| 6 | Schema validation failure |

## Summary of Decisions

| # | Decision | Rationale | Alternatives Considered |
|---|----------|-----------|------------------------|
| D1 | Binary is source of truth | Docs diverged speculatively; the running code represents the actual contract | Make docs authoritative and implement missing commands (too much scope) |
| D2 | Mark unimplemented as "Planned" | Preserves roadmap visibility without lying about current capabilities | Remove all unimplemented command docs (loses intent) |
| D3 | Fix `apx init app` YAML generation | Generated config must survive roundtrip through `config validate` | Document the incompatibility (breaks every new user) |
| D4 | Extend existing `doc_parity_test.go` | Already has patterns established; adding tests is incremental | New test file (unnecessary split) |
| D5 | Use sphinx `{admonition}` for "Planned" markers | Already supported by `sphinx_design` extension | Custom CSS class (harder to maintain) |
| D6 | Create stub pages for broken links | Sphinx warnings block CI; stubs are cheap | Remove toctree entries (loses structure) |
| D7 | One canonical exit-code table | Single source prevents drift across pages | Per-command exit codes (too granular) |
| D8 | Don't implement missing commands | Spec explicitly scopes this as docs + minimal code fixes | Implement list/show/update (scope creep) |
